# 灵羽 · POC 可行性验证文档

> 版本：对应产品形态 v3（飞书式所见即所得编辑器 + 半自动多平台填充）
> 文档性质：概念验证（Proof of Concept）。目的是回答一个问题——**这个产品值不值得继续投入，以及在投入前还需要先验证什么。**
> 立场：本文刻意偏向"找问题"而非"证明可行"。能被一份诚实的 POC 说服，才值得做。

## 0. 结论先行（TL;DR）

| 维度 | 判断 | 一句话理由 |
| --- | --- | --- |
| 效果可行性 | ✅ 已验证（核心链路） | WYSIWYG 编辑、图片往返、7 平台填充、发布前体检均已跑通；存储格式零污染。 |
| 工程可行性 | ⚠️ 可行但有存在性风险 | 技术栈成熟、测试/CI 齐全；但**依赖各平台 DOM 选择器，平台改版即失效**，维护成本随平台数 × 改版频率线性增长。 |
| 价值/市场可行性 | ❓ 未验证 | 目标人群真实存在但偏窄；"半自动"是安全护城河也是价值天花板；变现路径不清晰；当前仅 N=1（作者自用）验证。 |

**总体结论：有条件 GO。** 作为个人/小众工具，技术与效果已成立，可以继续做；但**在投入商业化或大规模推广之前，必须先用低成本手段验证"需求真实性"与"维护可持续性"**（见 §7、§9）。

## 1. 产品定义（当前形态）

**一句话定义**：灵羽是一个 Chrome MV3 扩展，帮内容创作者把"一篇 Markdown 长文 + 本地配图"半自动地填充到多个内容平台的原生编辑器里，**最终发布由人工在各平台确认**。

**三条不可让渡的核心约束**（既是产品定位，也是能力边界）：

1. **不调用平台私有发布 API**——只操作公开的网页编辑器。
2. **不替用户点击最终发布**——把内容送进编辑器，发布与否由人决定。
3. **本地优先**——任务、图片、设置存于浏览器 `storage.local`，不保存账号密码/Cookie，不上传历史。

**当前能力清单**：

- 飞书式所见即所得编辑器（底层 Markdown 存储），图片自动压缩、粘贴/拖入自动入库。
- 7 个自动填充平台：公众号、知乎、小红书、B 站专栏、人人都是产品经理、X、Reddit。
- 7 个快捷跳转平台：微博、简书、掘金、CSDN、今日头条、豆瓣、Medium。
- 平台短文案变体（小红书/X/Reddit），发布前体检（红/黄/绿），历史与备份，单篇导出 Markdown+图片 zip。
- 适配器当前页面自检（定位平台改版后失效的选择器）。

## 2. 目标用户与场景

### 2.1 用户分层

| 层级 | 画像 | 为什么是/不是目标 |
| --- | --- | --- |
| **主目标** | **Markdown 原生的技术/知识型创作者**（独立开发者、技术博主、AI/产品类公众号主），常年用 Typora/Obsidian/VS Code 写作，固定跨发到 3+ 平台 | 既有 Markdown 资产、跨发频率高、对"登录式自动化工具封号风险"敏感——三点全中 |
| **次目标** | **非技术但高频跨发的自媒体**（生活/职场/读书类），原本不写 Markdown | v3 的 WYSIWYG 编辑器**正是为打开这一层而做**——他们要的是"像 Word 一样写"，不接受看源码 |
| **边缘** | MCN/团队号矩阵运营 | 他们要的是"多账号 + 批量 + 排期 + 数据"，灵羽的"单人 + 半自动 + 不碰账号"与之错位 |
| **非目标** | 想"一键全自动群发"或"绕过审核"的用户 | 与三条核心约束直接冲突，不服务 |

> **关键洞察**：Markdown-first 把主目标收得很窄（多数公众号作者并不写 Markdown）；v3 的 WYSIWYG 改造是一次**有意识的扩圈**——用"所见即所得"把次目标人群纳入，同时保留 Markdown 作为底层资产。这是当前产品形态最重要的战略动作，也是本次 POC 需要验证其有效性的核心假设。

### 2.2 典型场景（JTBD：当我……我想……以便……）

- **S1 跨平台首发**：当我写完一篇长文，我想用最少的重复劳动把它发到公众号/知乎/掘金，以便一次创作覆盖多个渠道——**而不必在每个平台重排版、重传图**。
- **S2 配图搬运**：当我的文章有 8 张本地截图，我想避免"在每个平台的编辑器里逐张拖拽上传"，以便省下最枯燥的环节。
- **S3 平台特化**：当我要发小红书/X，我想从长文快速改出短文案 + 标签 + 话题，以便符合短内容平台的调性。
- **S4 安全焦虑**：当我考虑用"一键分发"类工具，我担心它要我登录账号、有封号风险，我想要一个**不碰我账号**的方案。
- **S5 发前自检**：当我准备发布，我想知道哪个平台还缺封面/缺变体/超字数，以便不在平台页面才发现问题。

### 2.3 痛点与现状替代方案

| 痛点 | 用户当前怎么做 | 现状的代价 |
| --- | --- | --- |
| 每平台重排版（尤其 Markdown→富文本） | 手动复制 + 平台编辑器里重调 | 每平台 5–15 分钟，纯体力 |
| 图片逐个平台重传 | 在每个编辑器里逐张拖拽 | 最枯燥、最易错（漏图/错序） |
| 短内容平台需另写文案 | 临时手写 | 容易敷衍，质量不稳定 |
| 跨发账号安全 | 要么手动、要么用登录式工具担风险 | 安全与效率二选一 |
| 不知道发到哪了/缺什么 | 脑记或表格 | 易漏发、易带错素材 |

> **诚实评估**：S1/S2/S5 是**真痛点（止痛药属性）**，对高频跨发者价值明确；S3 偏增强（维生素属性）；S4 是**差异化的情绪价值**，但只对"风险敏感型"用户成立。对低频创作者（一月发一两次），整体价值会退化为"省一点点事"，不足以驱动安装与留存。

## 3. 需求调研方式（如何验证"需求是真的"）

> POC 不能只验证"能做出来"，更要验证"有人真的需要"。以下方法按**成本由低到高 / 信号由弱到强**排列，建议组合使用。

| # | 方法 | 目标问题 | 样本/动作 | 产出 | 成本 |
| --- | --- | --- | --- | --- | --- |
| R1 | **Dogfooding（自用）** | 核心链路是否真的省事？哪里最爽/最烦？ | 作者本人连续 4 周用它发完所有文章，记录每篇耗时与卡点 | 真实耗时对比 + 卡点清单 | 极低 |
| R2 | **深度访谈** | 痛点是否普遍？现状替代方案是什么？愿不愿换？ | 5–8 位主/次目标用户，半结构化访谈各 30–45 分钟 | 痛点强度排序、JTBD 验证、是否"非它不可" | 低 |
| R3 | **竞品分析** | 别人怎么解、为什么没满足、定位空隙在哪 | 拆解蚁小二/易媒助手/简媒/融媒宝 + 各平台官方多账号工具 | 竞品能力矩阵 + 差异化空位 | 低 |
| R4 | **社群问卷** | 跨发频率分布、平台组合、对"登录式工具"的态度 | 在 V2EX/即刻/小红书创作者群/掘金 投放 100+ 份 | 量化的人群画像与意愿 | 中 |
| R5 | **着陆页 + 等待名单** | 有多少人愿意为"这句话"留邮箱？ | 一个讲清价值主张的着陆页，看转化率 | 价值主张的市场信号 | 中 |
| R6 | **灰度上架 + 埋点** | 真实留存与使用结构 | 上 Chrome Web Store，埋点：哪些平台被用、填充成功率、兜底复制使用率、7日留存 | 行为数据（最强信号） | 中高 |
| R7 | **平台政策调研** | 自动化/多发是否触线、长期合规风险 | 逐平台读 ToS + 社区封号案例 | 合规风险地图 | 中 |

**访谈问题清单（R2 示例，避免诱导性提问）**：
1. 你上一次把同一篇内容发到多个平台是什么时候？发了哪几个？
2. 整个过程哪一步最花时间/最让你烦？（让其复述，不给选项）
3. 你现在用什么工具或方法？为什么是它？
4. 你试过"一键分发"类工具吗？为什么用/不用？
5. 如果有个工具帮你填好但**仍要你自己点发布**，你会用吗？为什么？
6. （展示原型后）这里哪一点对你最有用？哪一点你不会用？
7. 这种工具值多少钱？你会为它付费吗，按什么方式？

> **关键原则**：R6（行为埋点）是唯一能戳破"嘴上说要、实际不用"的信号；R1–R5 用来决定"要不要做到 R6"。**当前产品只完成了 R1（且 N=1）**，这是本 POC 最薄弱的环节。

## 4. 效果可行性验证（Does it work & deliver value）

### 4.1 已验证的核心链路（实测）

- **WYSIWYG 编辑器**：浏览器实测——所见即所得渲染（标题/加粗/列表/图片/引用），编辑器即预览，单主区布局成立。
- **图片往返完整性**：图库"插入正文"→编辑器实时回灌；**存储 Markdown 零 `dataUrl` 泄漏**（字数仅 +17~26 而非 +数千，证明回写的是短 `filename` 引用）。
- **粘贴回归已修**：编辑器内新粘贴/拖入的 `dataUrl` 图自动入库并转 `filename`。
- **格式规范化可控**：Crepe 把缩放比例写进 `alt`（`![1.00](...)`）的污染已在映射层清洗；列表 `-`→`*` 等规范化已知并接受。
- **下游解耦验证**：变体/发布前体检/导出/适配器全部读 `task.markdown`，编辑器改造后**无需改动**——架构解耦成立。
- **7 平台填充**：合成粘贴 + `setNativeValue` + `setInputFiles` 路径已实现；公众号封面"从正文选择"等平台特例已处理。

### 4.2 效果指标定义与目标值（建议纳入埋点）

| 指标 | 定义 | 目标值 | 不达标意味着 |
| --- | --- | --- | --- |
| 填充成功率 / 平台 | 单次填充无报错且字段到位的比例 | 常态 ≥ 90% | 选择器失效或平台改版 |
| 单篇分发耗时降幅 | 用灵羽 vs 纯手动的耗时比 | ≥ 50% | 价值不足以驱动留存 |
| 兜底复制使用率 | "复制富文本"被使用的占比 | < 30% | 自动填充不可靠，退化成复制工具 |
| 图片零丢失率 | 分发后图片无漏/无错序 | ≥ 99% | 最枯燥环节没解决好 |
| 体检命中率 | 发布前体检拦下的真实问题数 | 定性观察 | 体检规则不贴实战 |

### 4.3 仍未验证的效果点（诚实清单）

- **跨平台真实成片率**：实测集中在 demo/单平台，**未做"一篇真文章一次性发完 7 平台"的端到端实测**（R1 应补）。
- **富文本保真度**：表格、代码块高亮、公式、脚注在各平台编辑器的落地效果未系统对比。
- **图片重排回归**：WYSIWYG 改造移除了预览区的"上移/下移"按钮，改依赖编辑器块拖拽，**该交互未验证 Crepe 是否顺手**。
- **大文档性能**：万字长文 + 几十张图在 Crepe 中的输入延迟未压测。

## 5. 工程可行性验证（Can we build & sustain it）

### 5.1 已验证项

- 技术栈：WXT + React 19 + TypeScript，成熟可控。
- 质量基线：**107 个单测通过、ESLint 干净、`npm run build` 成功（产物 4.75MB）、GitHub Actions CI（lint→test→build）**。
- 架构亮点：**Markdown 作为唯一序列化层**——编辑器只是"编辑表面"，下游逻辑全部解耦。这让"换编辑器"成为局部改动（v3 验证了这一点），是本项目最重要的工程资产。

### 5.2 ⚠️ 存在性风险：选择器脆弱性

这是**决定项目长期生死的单点风险**，必须正视：

- 填充依赖各平台编辑器的 DOM 选择器（`lib/adapters/<platform>.ts` 的 `SELECTORS`）。
- **平台前端改版 → 选择器失效 → 该平台静默失败**。这不是"如果"，而是"多久一次"。
- 维护成本模型：`维护工作量 ≈ 平台数 × 各平台改版频率 × 单次修复成本`。平台越多越不可持续；这与"多平台"的产品卖点**内在矛盾**。
- 缓解手段现状：已内置"当前页面适配器自检"降低定位成本，但**修复仍需人改源码 + 重新构建**——普通用户做不到。

**工程上的关键决策点（尚未解决）**：
- 选择器是否能**云端下发热更新**（用户不重装即可修复）？这需要一个轻后端，与"纯本地"定位冲突，是重大架构抉择。
- 若走 Chrome Web Store，扩展更新可自动推送，但**每次选择器修复都要过商店审核**，响应速度受限。

### 5.3 其它工程约束

- **MV3 限制**：Service Worker 生命周期、`content_scripts` 注入时机；部分平台用受控富文本组件，合成事件需 `setNativeValue` 等技巧绕过——已踩过，但每个新平台都要重新趟。
- **合成事件被拦截风险**：部分编辑器对非可信事件（`isTrusted=false`）有防护，可能导致填充无效。
- **包体积**：Crepe + KaTeX 字体使产物显著增大（4.75MB），对扩展可接受但非最优。
- **分发模型未定**：当前是"克隆源码自行构建"，门槛极高；要触达非技术用户，**必须上架 Chrome Web Store**——这又引入审核、隐私政策、自动化合规审查等新变量。

### 5.4 工程结论

**可行，但不是"做完就一劳永逸"的工程，而是一个需要持续养护的工程。** 选择器维护是长期负债，是否值得养护取决于 §6 的市场结论。

## 6. 价值 / 市场可行性验证（Should we build it）

### 6.1 市场规模（粗估，需 R4 校准）

> 以下为数量级推断，非精确数据，仅用于判断"是不是太小"。

- TAM：中文内容创作者数以千万计，但"高频跨发 + Markdown 友好 + 安全敏感"的交集很小。
- SAM（务实可服务）：技术/知识型独立创作者 + 愿接受 WYSIWYG 的轻度跨发者，**量级估计在十万级**。
- SOM（早期可获取）：通过开发者社群（掘金/V2EX/即刻）能触达的早期用户，**初期千级**已是乐观。

**判断**：这是一个**利基市场（niche）**，不是大众市场。适合做成"小而精的口碑工具"，不适合追求规模化增长曲线。

### 6.2 竞品矩阵

| 方案 | 自动化程度 | 账号安全 | 平台覆盖 | 价格 | 与灵羽的关系 |
| --- | --- | --- | --- | --- | --- |
| 纯手动复制粘贴 | 无 | 最安全 | 全 | 免费 | **真正的竞品**——灵羽必须明显优于它 |
| 蚁小二 / 易媒 / 简媒 / 融媒宝 等 | 高（多账号托管） | 有封号争议 | 广 | 多为付费 SaaS | 自动化更强但要托管账号；灵羽以"安全 + 本地"差异化 |
| 各平台官方多账号/同步工具 | 中 | 安全 | 仅自家生态 | 免费 | 不跨生态；灵羽跨平台 |
| Markdown→公众号 排版器（如 mdnice/135） | 无跨发 | 安全 | 单平台 | 免费/增值 | 只解排版不解分发；可被灵羽内化 |

> 竞品名单需在 R3 中核实其当前形态与定价，避免基于过时认知判断。

### 6.3 差异化与定位

灵羽的真正差异化是 **"安全 + 本地 + 半自动"** 的组合，而非"更强的自动化"。

- **护城河**：不碰账号 → 规避封号焦虑；开源 + 本地 → 信任成本低。
- **天花板**：正因为"仍要你自己点发布"，它**永远比不过托管式工具的极致省事**。安全是卖点，也是价值上限。

**定位建议**：不要试图和蚁小二比"自动化全面性"，而是占住**"给在乎账号安全、写 Markdown 的创作者的、最顺手的半自动跨发工具"**这个细分心智。

### 6.4 付费意愿与商业模式（诚实：变现难）

- 工具免费 + 本地，用户默认期待免费；直接收费阻力大。
- **可能的模式（均需验证）**：
  1. **纯开源 + 个人品牌/作品集**（已有 showcase 站）——不直接变现，价值在影响力。**当前最现实。**
  2. **Freemium：选择器自动热更新订阅**——把"最痛的维护"做成付费服务（"永远能用的适配器"）。逻辑自洽但需后端，与纯本地冲突。
  3. **一次性买断 Pro 功能**（AI 变体、排期、更多平台）——天花板低。
- **诚实判断**：短期不应以变现为目标；先把它当**作品/工具 + 需求探针**，用 R6 数据决定是否值得为模式 2 投入后端。

### 6.5 价值结论

**价值对窄人群真实成立，但市场规模有限、变现路径未验证。** 适合作为高质量利基工具/个人作品推进；不适合按"创业项目找 PMF 再融资"的预期推进，除非 R4/R6 给出意外强的信号。

## 7. 落地与迭代方案

> 原则：**用最低成本先验证最不确定的假设**（需求真实性 > 维护可持续性 > 功能丰富度）。

### P0 · 当前（已完成）
核心填充 + 安全定位 + WYSIWYG + 质量基线（测试/CI）。**已具备 dogfooding 与小范围分发条件。**

### P1 · 验证期（下一步，2–4 周，重点不是加功能）
**目标：把 N=1 变成 N=10，拿到真实使用信号。**
- [ ] R1 端到端实测：作者用它真实发完 4 周内全部文章，记录每平台成功率与耗时（补 §4.3 缺口）。
- [ ] R2 访谈 5–8 人，验证主/次目标人群与 WYSIWYG 扩圈假设。
- [ ] 上架 Chrome Web Store（或内测分发），接入**最小埋点**：平台填充成功率、兜底复制使用率、留存。
- [ ] 补回图片"上移/下移"按钮式交互（已知回归，低成本，提升顺手度）。
- [ ] R7 平台政策快速扫描，标出高风险平台。

**P1 验收门槛（决定是否进 P2）**：≥ 5 位非作者用户完成一次真实跨发；填充成功率均值 ≥ 85%；至少 3 人表示"会继续用"。

### P2 · 增强期（仅当 P1 通过）
- [ ] 选择器韧性工程：失效自动探测 + 友好降级提示 + （评估）云端热更新后端。
- [ ] AI 短文案变体升级（接入模型，质量与可控性）。
- [ ] 富文本保真度专项（表格/代码/公式逐平台对比修复）。
- [ ] 大文档性能压测与优化。

### P3 · 商业化探针（仅当 P2 留存成立）
- [ ] 验证"适配器热更新订阅"付费意愿（着陆页 / 小范围灰度定价）。

## 8. 深刻反思（Risks & Honest Critique）

1. **选择器税是悬顶之剑。** 产品越成功（平台越多、用户越多），平台改版导致的静默失败就越伤口碑。这是与"多平台"卖点内在矛盾的结构性风险。**不解决热更新，规模化即埋雷。**

2. **N=1 验证陷阱。** 目前唯一被验证的用户是作者本人，作者既是开发者又是 Markdown 重度用户——**最不具代表性的样本**。"我觉得有用"不等于"市场需要"。P1 的访谈与埋点是戳破这层泡沫的唯一办法。

3. **维生素还是止痛药，取决于频率。** 对高频跨发者是止痛药，对低频者是维生素。而**愿意装一个扩展、学一套工具的人，往往恰恰是低频尝鲜者**——目标用户与早期获取用户可能错位。

4. **Markdown-first 的双刃。** 它精准锁定技术创作者（优点：人群清晰、付费能力相对强），也天然排除多数公众号作者（缺点：池子小）。v3 的 WYSIWYG 是对冲，但**对冲是否成功、还是两头不讨好（技术用户嫌它重、非技术用户嫌它仍是开发者工具），需要 R2/R6 给答案**。

5. **"半自动"的价值天花板被低估了风险。** 安全定位很美，但用户的真实诉求可能是"我就想全自动省心"。如果访谈发现"安全"在用户排序里靠后、"省事"靠前，那么灵羽的差异化护城河就是在保护一个用户没那么在乎的东西。

6. **Build vs Buy。** 在重投入前应认真回答：我是在做一个比蚁小二"自动化更弱、但更安全"的工具——这个 trade-off，市场买不买账？还是说我其实是在为"作品集 + 自用"做一个足够好的个人工具？**这两个目标对应完全不同的投入强度，必须先想清楚。**

7. **技术债与已知回归。** WYSIWYG 改造带来：图片插入只能到文末（非光标）、图片重排依赖拖拽、包体增大。这些是为"扩圈"付的合理代价，但**不应在没验证扩圈成功前继续加重**。

## 9. 验收结论

### 9.1 三维度评分

| 维度 | 评分 | 依据 |
| --- | --- | --- |
| 效果可行性 | 8 / 10 | 核心链路实测通过、架构解耦干净；扣分在跨平台端到端实测与保真度未补齐。 |
| 工程可行性 | 6 / 10 | 基线扎实、可持续构建；扣分在选择器维护是长期负债、热更新未解、分发模型未定。 |
| 价值/市场可行性 | 4 / 10 | 痛点对窄人群真实；扣分在 N=1 验证、市场偏小、变现未验证。 |

### 9.2 总体结论

**有条件 GO。**

- 作为**个人工具 / 作品集项目**：✅ 直接通过，继续打磨即可，它已经是一个可用且有品味的产品。
- 作为**面向市场的产品**：🟡 暂缓重投入。**先完成 P1 验证（R1 端到端 + R2 访谈 + R6 埋点）**，用真实信号回答"需求是否普遍、扩圈是否成功、维护是否可持续"三问，再决定是否进入 P2/P3。

### 9.3 验收 Checklist（进入下一阶段的硬门槛）

- [ ] 作者完成连续 4 周真实 dogfooding，并产出耗时对比数据。
- [ ] ≥ 5 位**非作者**用户完成至少一次真实多平台分发。
- [ ] 7 平台填充成功率均值 ≥ 85%，兜底复制使用率 < 30%。
- [ ] 完成 5–8 份用户访谈，验证（或证伪）"WYSIWYG 扩圈"假设。
- [ ] 完成竞品定价/形态核实与平台政策风险地图。
- [ ] 明确选择器热更新的技术路线决策（做/不做/怎么做）。

**满足以上 ≥ 4 项且无致命否决信号 → 进入 P2；否则维持"个人工具"定位，停止市场化投入。**
