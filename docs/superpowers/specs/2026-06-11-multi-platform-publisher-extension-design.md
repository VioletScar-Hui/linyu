# 多平台文章分发 Chrome 插件 — 设计文档

日期：2026-06-11
状态：已与用户确认

## 1. 背景与目标

用户日常用 Markdown 写作（文章 + 本地配图），写完后需要逐个平台手动排版发布，重复劳动多。本插件目标：写完文章后，通过插件把文章和配图**半自动填充**到各平台的编辑器中，用户人工检查后点击平台自身的发布按钮完成发布。

**核心原则：**

- 半自动，不全自动：插件只负责"填好"，发布动作由用户完成。不调用平台私有接口，注入走编辑器的正常粘贴/上传通道，风控风险最低。
- 任何一步失败都有手动兜底，用户永远不会被卡住。

## 2. 范围

### V1（本期）

| 平台 | 能力 |
|---|---|
| 微信公众号 | 自动填充：标题、摘要、正文（含图）、**封面图** |
| 知乎 | 自动填充：标题、正文（含图） |
| 人人都是产品经理 | 自动填充：标题、正文（含图） |
| 小红书 | 自动填充：配图、标题（≤20 字）、正文变体（≤1000 字，含话题） |
| B站专栏 / X / Reddit | 仅 popup 快捷跳转到发布页 |

其他 V1 功能：撰写页（Markdown 导入、图片匹配、预览、小红书变体编辑、封面指定）、历史任务列表（最近 20 篇）、转换管线、失败时复制富文本兜底。

### V2（不在本期）

- X / Reddit 摘要导流短文自动填充（数据模型已预留"平台变体"字段）
- B站专栏适配器
- 发布结果追踪

## 3. 架构

技术栈：**TypeScript + WXT**（MV3 插件框架，自动生成 manifest、热重载）+ **React**（撰写页 UI）+ **markdown-it**（转换）+ **Vitest**（单测）。

权限：`storage`、`unlimitedStorage`、`tabs`、`scripting`；host 权限仅限目标平台域名。

### 组件

1. **Popup**：7 个平台快捷跳转按钮 + "新建分发"按钮（打开撰写页）。
2. **撰写页 Composer**（插件内独立标签页，React）：
   - 粘贴 Markdown 或导入 .md 文件；
   - 拖入本地图片，按文件名自动匹配 Markdown 中的图片引用，缺图标红；
   - 实时预览；
   - 封面图指定（默认第一张配图）；
   - 小红书变体编辑区（标题/正文自动生成初稿，可改）；
   - 每平台一个"去发布"按钮 + 状态灯（待填充 / 已填充待检查 / 失败 + 原因）；
   - 历史任务列表：最近 20 篇，含各平台状态，可重开继续分发。
3. **Background Service Worker**：任务存储（chrome.storage.local）、打开平台标签页并绑定"标签页 → 任务"映射、消息路由。
4. **平台适配器**（content scripts，每平台一个模块），统一接口：
   ```
   checkLogin() → fillTitle() → fillContent(html) → insertImages(files) → report(status)
   ```
   注入手法：构造合成 paste/drop 事件（DataTransfer 携带 text/html 与图片 File），派发给平台编辑器，由编辑器自身上传逻辑接管图片（使用用户已登录会话上传至平台图床/素材库）。
5. **转换管线**（纯函数库，独立模块）：Markdown→HTML、图片引用解析与文件名匹配、小红书文案截断、封面提取。

### 数据模型

分发任务（存 chrome.storage.local）：

```
Task {
  id, createdAt, title,
  markdown,                 // 原文
  images: { filename, base64, size }[],
  coverFilename,            // 封面图（默认第一张）
  variants: {               // 平台变体，V2 的 X/Reddit 复用此结构
    xiaohongshu: { title, body }
  },
  platformStatus: { [platform]: pending | filled | failed(reason) }
}
```

历史保留最近 20 篇，超出按时间淘汰。

## 4. 数据流

1. 撰写页：粘贴 Markdown + 拖图 → 解析引用、匹配文件名 → 保存任务。
2. 点"去发布 · 某平台" → background 打开该平台编辑页，记录 tabId → taskId 映射。
3. 适配器加载 → checkLogin（未登录则回报"请先登录"，支持登录后重试）→ 填标题 → 粘贴正文 HTML（图随正文注入，平台自动上传）→（公众号）填摘要、注入封面 →（小红书）传图、填变体文案。
4. 适配器确认编辑器内容真实存在后回报"已填充" → 撰写页状态灯变绿 → 用户人工检查、点平台发布按钮。
5. 各平台标签页互不影响，可并行。

## 5. 各平台适配要点

- **知乎**（最易）：ProseMirror 编辑器，HTML 粘贴友好；标题为独立输入框；图片随粘贴自动上传。
- **微信公众号**（最难）：编辑器位于 iframe（UEditor），content script 需 `all_frames` 进入 iframe；标题、摘要、正文分别填充；**封面图**：撰写页指定封面 → 适配器通过封面上传入口的 `input[type=file]` 注入文件（DataTransfer 设置 files + 触发 change）；若实测该路径被平台限制，降级为自动触发"从正文选择封面"并提示用户确认一步。封面自动化是 V1 风险最高项，需优先实测。
- **人人都是产品经理**：富文本编辑器，HTML 粘贴方案可行；具体 DOM 选择器开发时实测确定。
- **小红书**：创作服务平台"发布笔记"，流程为先传图后写文；适配器将配图设置到上传框（input.files），再填标题与变体正文。

适配器代码规范：所有 DOM 选择器集中在各适配器文件顶部常量区，平台改版只改一处。

## 6. 错误处理

- **未登录**：回报"请先登录"，登录后可点"重试填充"。
- **DOM 找不到 / 平台改版**：单步超时 10 秒即报失败，撰写页显示失败步骤，并提供**"复制富文本"按钮**（转换好的内容入剪贴板，手动粘贴兜底）。
- **图片问题**：解析阶段即发现 —— 缺图标红、单图 >10MB 警告，不拖到填充阶段。
- **状态真实性**：仅在适配器确认编辑器内有内容后才报"已填充"，不乐观假设。

## 7. 测试策略

- 转换管线为纯函数，Vitest 单测覆盖（Markdown 转换、图片名匹配、小红书截断、封面提取）。
- 适配器面向真实网站，无法自动化：每个适配器配手动验收清单（登录检测 / 标题 / 正文格式 / 图片 / 封面 / 失败兜底），开发完成后逐项实测。
- 平台改版属常态，选择器集中管理以降低维护成本。

## 8. 已确认的决策记录

| 决策 | 选择 |
|---|---|
| 自动化深度 | 半自动填充（非全自动、非剪贴板） |
| 技术路线 | 纯 MV3 插件自研适配器（不依赖本地服务，不基于开源二开） |
| 文章来源 | Markdown 文本 + 本地图片文件 |
| X/Reddit 内容形态 | 摘要导流短文（V2 实现填充，V1 仅跳转） |
| V1 适配器 | 公众号（含封面）、知乎、人人都是产品经理、小红书 |
| 历史记录 | 保留最近 20 篇分发任务 |
